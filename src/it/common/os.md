> Здесь кратко будут указаны фичи ОС или их механизмы...

- Windows оперирует Win32 API и NT, Win32 - чуть более высокоуровневая обёртка над NT
- Linux оперирует POSIX API или NTPL (Native POSIX Thread Library)
# Threads, processes
- Напомню, что каждый **поток "сидит" на heap, data, code процесса**.. но **у потока всегда свой stack** (ну и thread local storage)
- **flock** — это _файловая блокировка_. Работает на уровне **дескрипторов файлов** и используется для синхронизации _разных процессов_ (**IPC !!!**), не для синхронизации потоков внутри одного процесса
- При **наследование файловых дескрипторов** - **оба процесса получают общую позицию в файле**
	- есть методы, где указывается, с какой позиции читать...
		- родительский на позиции 100
		- дочерний на позиции 100 и читает 20 символов => дочерний теперь на позиции 120
		- родительский теперь тоже на позиции 120 стоит
## Windows
- Поток - `CreateThread()`, процесс - `CreateProcess()`
- В `CreateProcess()` можно указать, чтобы дочерний процесс наследовал все handle'ы [link](https://learn.microsoft.com/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessa#:~:text=If%20this%20parameter%20is%20TRUE,of%20inheritable%20handles%2C%20see%20Remarks.)
## Linux
- Поток - `pthread_create()`, процесс - `fork()`
- В `fork()` страницы памяти родителя копируются только при записи (Copy-on-Write) => **т.е. у каждого процесса своё адресное пространство** => write в дочернем не повлияет на родительский
	- **мьютекс** при вот таком **копировании страниц памяти**, к сожалению, **по документации** оказывается **"в непонятно каком состоянии**"
- При вызове `fork()` дескрипторы файлов копируются, а указатель позиции в файле общий => **дочерний получает те же права на запись и чтение файла, что и родительский**
- Не все примитивы синхронизации хорошо переносимы...
	- `pthread_mutex_lock` - не на всех Unix поддерживается
	- `flock`, `semap` - стандарт POSIX.1-2001+
- список семафоров можно получать через `ipcs -s`
- [A bit about linux forks logic](linux-forks.md)

## Linux

# Signals
## Windows
- Прям по изначальной задумке их нет, есть имитация POSIX-сигналов.
- Целиком за подобного рода контекст отвечает **Structured Exception Handling (SEH)**
## Linux
- целиком **асинхронная модель** уведомления процессов
	- поэтому стоит быть аккуратными с примитивами синхронизации и особенно condition variable
- отправка сигналов - `kill()`, установка обработчиков на процесс - `signal` и `sigaction`
- список сигналов и назначение - [link]([https://man.archlinux.org/man/signal.7.ru#:~:text=sensible%20software%20design!](https://man.archlinux.org/man/signal.7.ru#:~:text=sensible%20software%20design!).-,%D0%A1%D1%82%D0%B0%D0%BD%D0%B4%D0%B0%D1%80%D1%82%D0%BD%D1%8B%D0%B5%20%D1%81%D0%B8%D0%B3%D0%BD%D0%B0%D0%BB%D1%8B,-Linux%20supports%20the))
	- *SIGTERM* - graceful shutdown
	- *SIGKILL* - instant die
- *SIGKILL*, *SIGSTOP* - нельзя заблокировать/переопределить/игнорировать
- если **не установлен обработчик сигнала** - **вызывается стандартное** для него дефолт действие

# IPC
- Обе операционки умеет в **shared memory (shm)**
	- это **некоторая общая память для всех процессов**, обычно **доступ к ней по имени** (которое сам же и задаёшь при создании)
	- если мы хотим **подгрузить** туда ещё какой-то **буффер свой** - то обычно ещё делают **маппинг** на него
	- **размер shm обычно фиксирован** (задаётся при создании)
## Windows
### shm
- делается слегка обходным путём через `CreateFileMapping()` и указать внутри `INVALID_HANDLE_VALUE`, ну и имя потом...
### RPC
> `Remote Procedure Call` некоторый синхронный внутренний сервер. Только линки могут быть как "процесс-процесс", "юзер-юзер" и т.п. Устарело*
- *link: https://csandker.io/2021/02/21/Offensive-Windows-IPC-2-RPC.html*

### ALPC (and LPC)
> `Local Procedure Call` - асинхронный вариант RPC
> `Advanced LPC` (иногда 'A' относится с асинхронному...) - оч шустрый механизм IPC передачи данных
- фича с асинхронности и очень быстрой передачи данных
	- объекты больше 4кБ вроде (размер можно менять) передаются через shared memory. Отдаётся только указатель на них. (aka zero-copy или map-data вместо copy) 
	- объекты до 4кБ просто быстро копируются на уровне kernel mode
- механизм никак не задокументирован.. его раскапывают, юзают и объясняют разработчики самостоятельно
- *link: https://csandker.io/2022/05/24/Offensive-Windows-IPC-3-ALPC.html*
### Misc
- `CreatePipe()` однонаправленный пайп
- `CreateNamedPipe()` однонаправленный именованный пайп
- `WSASocket()` berkley sockets (с Win10 добавили подобие *AF_UNIX*)
- `CreateMailslot()` однонаправленный broadcast
## Linux
- сокет тоже представляет из себя inode ссылку, как и любая абстракция в ОС
- `close()` - уменьшает ref count на 1 (сокет закроется, если ref count будет равен 0)
- `shutdown()` - немедленно разрывает TCP соединение и для всех процессов, ссылающихся на этот сокет, он закроется
### shm
- создать: `shm_open("my shared memory", flags, O_CREAT)`
- почитать: `shm_open("my shared memory", flags, O_RDONLY)`
- `mmap()` - для маппинга своего буффера
- (хак) обходным путём *менять размер shm* через `ftruncate` и повторный `mmap` для всех процессов  *можно, но не стоит...*
### Unix domain sockets (UDS)
> (*или AF_UNIX*) Это вот прям сокеты (bidirectional), только общение через прям файловую систему, где адресами являются имена/пути в файловой системе.
- это сильно быстрее сетевой части ОС из-за отсутствия всяких роутингов, проверок чексум и прочих сетевых особенностей
- это вот прям сокеты => bidirectional, data stream или datagrams
- требует ручного `unlink()` для закрытия
### Misc
- `pipe(int pipefd[2])`, где `[0]` - пайп на чтение, а `[1]` пайп на запись
	- `pipe2(...)` - то же самое с флагами
- `mkfifo()` создаёт FIFO (first in first out) именованный пайп
- `eventfd()` создаёт файловый дескриптор, который реагирует на события
- `signalfd()` создаёт файловый дескриптор, который реагирует на сигналы
# Misc
> Я хз под что это классифицировать и где понадобится, но...
-  `select()` - глобально нужен для "пройдись по всем дескрипторам, что я тебе дал и верни первый готовый"
	- это прям как `select` в *Go* для каналов, там это звучит как "выдай мне первый готовый канал"
 - `poll()` - глобально это *better select()* за счёт отсутствия ограничения к размерам списка, сами списки задаются удобной структурой
 - `epoll()` - глобально это *better poll()* за счёт обхода *красно-чёрным деревом вместо O(n)* и тем. Оно под капотом формирует во время обхода список готовых дескрипторов, которые получаются из `epoll_wait()`
- **select() вместо sleep():** Используется для **микросекундной задержки** (`struct timeval`), в то время как `sleep()` — секунды. Также `select()` позволяет не блокировать сигналы (может быть прерван EINTR), в отличие от `usleep` в старых реализациях.
- `fanotify()` позволяет перехватывать события файловой системы, оно реализовано прям на уровне ядра

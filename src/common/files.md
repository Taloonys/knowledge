# General
> **Файл** – в общем и целом.. некоторая абстракция ОС, которая всегда содержит указатель на блок данных на физическом диске
> **Директория (папка, каталог )** – совокупность файлов... на линуксе это список для маленьких директорий и HTree для больших, на винде это Btree
> **Файловой системой** называется функциональная часть ОС, обеспечивающая выполнение операций над файлами. 
> Примерами файловых систем являются FAT (FAT – File Allocation Table, таблица размещения файлов), NTFS (любимчик Windows), UDF (используется на компакт-дисках) и т.п.

- когда делаем `open()` на файл, то **нам возвращается** некоторый id -> **handle/дескриптор**, за счёт которого мы ссылаемся на тот или иной объект файловой системы (ну и сокетов, сессий и т.п.)
  ```
  Пользовательская программа                    Ядро ОС
       │                                         │
       │  open("file.txt")                       │
       ├────────────────────────────────────────►│
       │                                         │  Ядро:
       │                                         │  1. Находит inode/MFT-запись
       │                                         │  2. Проверяет права
       │                                         │  3. Создает объект "открытый файл"
       │                                         │  4. Помещает его в таблицу процессов
       │                                         │
       │  возвращает 0x0000007B (handle/fd)      │
       │◄────────────────────────────────────────┤
       │                                         │
       │  read(0x0000007B, buffer, 100)          │
       ├────────────────────────────────────────►│
       │                                         │  Ядро:
       │                                         │  1. Смотрит в таблицу процесса
       │                                         │  2. Находит объект по ключу
       │                                         │  3. Читает данные из реального файла
  ```
  -  с `write()` происходит запись данных в **буффер** записи и чтения из файла, но ещё **не в файл**! 
	  - **Буффер один**. 
	  - Размер буфера декларируется особенностями конректного “инструмента” работы с файлами и типа файловой системы (может зависеть от размера сектора… 512 байт или 4 Кбайта…). 
	- **Запись из буфера в файл** происходит в трёх случаях:
		- При вызове **flush()** - принудительная запись из буфера в файл
		- При закрытии файла через **close()**
		- При переполнении буфера

> Если файл открыт на запись, то другие программы не могут ничего в него записать, если открыт на чтение - то все могут читать. Существование буфера обусловлено оптимизацией и уменьшением кол-ва обращений к диску в виду его затратности и ресурсу носителя. 

>💡 Некоторые файловые системы ведут *журналирование* - определённый журнал с удалением и перемещение файлов. Если на диски с таким устройствам часто осуществлять перезапись, то это довольно сильно исчерпывает ресурс диска.
# Windows
> Родной формат `ntfs`
- дурной тем, что **легко фрагментируется по диску**...
- windows иногда *пытается делать дефрагментацию только  для HDD*
- **case insensitive** касательно имён файлов
- умеет в шифрование на лету
- журналирует целиком метаданные и данные файла
```
Файл → Master File Table (MFT) record
       ├── стандартная информация (времена, права)
       ├── имя файла (может быть несколько потоков, например, "file.txt:stream")
       └── указатели на данные
```
# Linux
> Родной формат `ext4`
- использует *extents* - aka **старается делать непрерывные блоки памяти**
	- .. соответственно, фрагментация на диске минимальна. Дефрагментация примерно никогда не нужна
- модель прав/доступа строится на разделение группа/владелец/остальные и соответствующие права для этих лиц
- **case sensitive** касательно имён файлов
-  журналирует, в основном, метаданные файла
- [Про именованые каталоги](https://wiki.merionet.ru/articles/obyasnenie-struktury-katalogov-linux)
## Inode
- каждый файл = "имя" + "ссылка на inode"
	- имя файла хранится в некотором отдельном каталоге (например, `/dev/sda`) примерно так:
	  ```
	  file_name | inode | hardlinks
	  ----------+-------+-----------------
	  README.md | 41231 | jump_to_docs.txt
	  ```
	- файл иногда называют *directory entry*
### links
> специальные файлики, для прыжка на другие нужные нам файлы
- **symlink (softlink)** -> отдельный файл со своей inode и именем со строкой с путём до таргет файла
	- если путь короткий, то хранится прям в `i_block` inode'ы, а не на диске
	- если удалить файл под симлинком, то симлинк будет битым
- **hardlink** - см. выше -> это буквально второе имя файла с полностью такими же правами и природой
- если прыгаем по симлинкам больше 40 раз, то ядро вернёт ошибку.. дабы не упасть в бесконечный цикл...

> *inode или Index node* содержит:
- size
- права доступа
- timestamps
	- когда создан
	- когда модифицирован
	- когда менялись метаданные
- `i_ilinks_count` -> счётчик жёских ссылок
- `i_block` -> указатель на блок данных на диске

## File delete
- происходит `unlink()`, но файл сам по себе ещё не удаляется
- из файловой системы просто пропадает имя файла
- файл прям удаляется, если счётчик ссылок равен 0 ~ нет открытых дескрипторов
